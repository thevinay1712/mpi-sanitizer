MPISan: Compiler-Integrated MPI Usage Sanitizer

A Layman's Guide for Project Evaluation

1. The Big Picture: What is MPI and What is MPISan?

MPI (Message Passing Interface) is the standard way supercomputers run code. Instead of one computer doing all the work, MPI breaks the code into "Ranks" (separate processes or computers) that talk to each other over a network to solve huge problems.

The Problem: When 100 computers are talking to each other at the same time, programmers make mistakes. They send data to the wrong place, overwrite memory, or wait for messages that never come. These bugs are nearly impossible to find because the code doesn't explicitly crash—it just silently gives the wrong answer.

The Solution (Our Project): We built MPISan. It is a "Sanitizer"—a security guard that lives inside the compiler. It watches every single message being sent and received, and if it detects a rule being broken, it immediately crashes the program and points exactly to the error.

📬 The Analogy: Imagine MPI is a global postal service. Millions of letters are moving around. MPISan is a network of invisible inspectors embedded in every post office. They check if the addresses match, if the envelopes are too small for the contents, and if someone is expecting a letter from themselves.

2. How It Works: The Flowchart

Our project consists of two major moving parts working together. Below is a flowchart showing exactly what happens from the moment we compile the code to the moment a bug is detected:

Part A: The LLVM Compiler Pass (MPISanPass.cpp)

What it is: A plugin for the Clang/LLVM compiler.

What it does: When you compile your C++ code into machine code, our pass sneaks in and scans your code for any MPI function (like MPI_Send). Before it finishes compiling, it literally rewrites your code to add our secret tracking functions (like __mpisan_check_send) right next to the original ones.

Part B: The Runtime Library (mpisan_rt.cpp)

What it is: A background library that runs alongside the actual application.

What it does: When the program runs, those "GPS trackers" we installed wake up. They send data to our Runtime Library. The runtime library does the math, tracks memory, and communicates with other computers using a secret "Shadow Communicator" to verify if the rules are being followed.

3. How to Run the Project

To demonstrate this to your professors, you can show them exactly how easy it is to use. The user does not need to change a single line of their C++ code. They just use our compiler wrapper.

# 1. Compile the code using our custom MPISan wrapper instead of normal mpicxx
./mpisan-cxx.sh tests/test_05_buffer_aliasing.cpp -o test_05

# 2. Run the application across 2 simulated computers (ranks)
mpirun -np 2 ./test_05

4. The Bugs We Catch (With Analogies)

Bug 1: Buffer Aliasing (Overlapping Memory)

The Bug: In asynchronous MPI (non-blocking), the computer starts sending data in the background while the code keeps running. If the programmer accidentally tells the computer to receive new data into the exact same memory space before the send is finished, the data gets corrupted.

📝 The Analogy: Imagine two people trying to write a letter on the exact same piece of paper at the exact same time. It results in unreadable scribbles. We catch this by assigning strict ownership to pieces of paper until the writing is done.

Bug 2: Mismatched Collectives

The Bug: "Collectives" are team meetings where every computer must participate (e.g., MPI_Bcast). If 9 computers show up for the meeting, but the 10th computer skips it, the other 9 computers will wait forever, freezing the supercomputer.

📅 The Analogy: Imagine a group project where everyone must sign the attendance sheet. At the end of the semester, the teacher (MPISan) checks the sheets. If Bob only signed 5 times but Alice signed 6 times, the teacher knows someone skipped a mandatory meeting!

Bug 3: Buffer Overflow

The Bug: Computer A sends 100 bytes of data, but Computer B only prepared a bucket that holds 50 bytes. The extra 50 bytes spill over and corrupt the rest of Computer B's memory.

📦 The Analogy: You try to stuff a giant microwave into a tiny P.O. Box. The post office breaks the box trying to make it fit. We prevent this by having the sender use our hidden "Shadow Communicator" to call the receiver in advance and ask: "How big is your box?"

Bug 4: Deadlocks (Sending to Self)

The Bug: A computer uses a blocking MPI_Send to send a message to itself. Because it is blocked waiting for the send to finish, it can never reach the MPI_Recv code to actually receive the message. It is stuck forever.

📱 The Analogy: You call your own phone number and wait on the line for yourself to pick up. You will wait forever. We act as the phone company and instantly block the call, telling you it's a mistake.

5. The Final Proof: The LULESH Evaluation

To prove our project isn't just a toy, we ran it against LULESH, an Exascale proxy application created by the Lawrence Livermore National Laboratory. LULESH simulates real-world physics (hydrodynamics) and contains thousands of lines of highly complex, optimized MPI code.

The Result: Our compiler wrapper successfully instrumented the massive codebase, tracked the memory natively, and allowed the simulation to run to completion across 8 processing ranks without a single false positive.

Why this is impressive: It proves our tool is scalable, robust, and mathematically sound enough to handle real-world Supercomputing software without breaking it.
